# Init Bootstrap: Config Defaults, DevOps Scaffold, and Create-Pipeline UI

## Problem

New users running `kaos-control init` for the first time must land in a fully functional state without manual config authoring. Today the init scaffold and generated `config.yaml` may omit default blocks for the idea-capture agent, kanban view, and dashboard — forcing users to consult documentation and hand-edit YAML before the tool is usable. Additionally, the project scaffold does not create a `devops/` directory, and the existing DevOps screen lacks a "Create Pipeline" workflow, meaning users cannot author new pipeline definitions from the UI.

## Goals / Non-goals

### Goals

- **G1** — `kaos-control init` produces a `config.yaml` that is immediately usable: the generated file must include sensible default blocks for the idea-capture agent, kanban columns, and dashboard configuration.
- **G2** — The project scaffold creates a `devops/` directory alongside the existing `lifecycle/` stage directories.
- **G3** — The DevOps screen provides a "Create Pipeline" action that allows users to author a new pipeline definition using the existing YAML editor component (CodeMirror-based), maintaining UI consistency.
- **G4** — The onboarding path from `init` to first productive use requires zero manual file editing.

### Non-goals

- Modifying the behaviour of existing pipelines or pipeline execution.
- Adding pipeline template libraries or marketplace features.
- Changing the app-level config (`~/.kaos-control/config.yaml`) — this requirement covers the per-project `lifecycle/config.yaml` generated by init.
- Redesigning the DevOps screen layout beyond adding the create-pipeline workflow.

## Detailed Requirements

### Functional

#### FR-1: Config template defaults

The `config.yaml.tmpl` template used by `kaos-control init` must emit default configuration blocks for:

1. **Idea-capture agent** — an agent entry with `name: idea-capture`, `driver: inline`, a `prompt_templates.idea-capture` block, and `allowed_write_paths: [lifecycle/ideas]`. The generated prompt template must be functionally equivalent to the one in this project's own `lifecycle/config.yaml`.
2. **Kanban view** — a `kanban:` block with default columns mapping lifecycle statuses (at minimum: Backlog → `[draft]`, In-Progress → `[clarifying, planning, in-development, in-qa]`, Done → `[done, abandoned, rejected]`) and default `card_fields`.
3. **Dashboard** — a `dashboard:` block with `tracked_types` defaulting to `[idea, requirement, defect]`.

The template must use Go `text/template` syntax consistent with the existing template infrastructure in `internal/initcmd/`.

#### FR-2: DevOps directory scaffold

`scaffoldDirs()` (or its equivalent) must create a `devops/` directory at the project root during `kaos-control init`. If `devops/` already exists, init must not fail or overwrite its contents.

#### FR-3: Create Pipeline UI

The DevOps screen (`DevOpsView.vue`) must include a "Create Pipeline" action (button or equivalent affordance) that:

1. Opens a form/dialog containing the existing YAML editor component (CodeMirror 6) for authoring a pipeline definition.
2. Validates that the entered YAML is syntactically valid before allowing submission.
3. On submission, writes the pipeline definition to the `devops/` directory via an appropriate API endpoint.
4. After successful creation, the new pipeline appears in the pipeline list without requiring a page refresh.

#### FR-4: Backend API for pipeline creation

If no existing endpoint supports creating a new pipeline definition file, add a `POST /api/p/{project}/devops/pipelines` endpoint that:

1. Accepts a JSON body with at minimum `slug` (string, validated against the slug pattern) and `definition` (string, valid YAML).
2. Writes the definition to `devops/{slug}.yaml` within the project directory.
3. Returns `201 Created` with the created pipeline metadata.
4. Rejects duplicate slugs with `409 Conflict`.
5. Enforces the same role restrictions as existing DevOps endpoints (`product-owner`, `devops`).

### Non-functional

- **NF-1**: The init command must remain idempotent — running `init` on an already-initialised project must not corrupt or overwrite existing config or `devops/` contents (unless `--force` flags are used).
- **NF-2**: The YAML editor in the create-pipeline dialog must provide syntax highlighting and basic error feedback consistent with the existing editor experience elsewhere in the app.
- **NF-3**: The create-pipeline workflow must be accessible only to users with `product-owner` or `devops` roles, matching existing DevOps screen access control.

## Acceptance Criteria

- [ ] Running `kaos-control init` in a fresh directory produces a `lifecycle/config.yaml` containing an `idea-capture` agent block, a `kanban:` block with default columns, and a `dashboard:` block with default `tracked_types`.
- [ ] Running `kaos-control init` creates a `devops/` directory at the project root.
- [ ] Running `kaos-control init` twice does not error, overwrite existing config, or duplicate the `devops/` directory.
- [ ] The DevOps screen displays a "Create Pipeline" button visible to users with `product-owner` or `devops` roles.
- [ ] Clicking "Create Pipeline" opens a dialog containing the CodeMirror YAML editor.
- [ ] Submitting a valid pipeline definition via the dialog creates a file in `devops/` and the pipeline appears in the list immediately.
- [ ] Submitting invalid YAML displays a validation error and does not create a file.
- [ ] Submitting a duplicate slug returns an appropriate error message.
- [ ] The generated config template passes `go build` and the init integration tests pass.
- [ ] Related: [[kaos-control-init-bootstrap]]

## Open Questions

1. **Pipeline definition schema** — Is there a defined schema or required structure for pipeline YAML files in `devops/`? If so, should the create-pipeline form enforce it, or is free-form YAML acceptable for v1?
2. **Seed pipelines** — Should `kaos-control init` also generate any example/seed pipeline definitions in `devops/`, or should it start empty?
3. **Config merging on re-init** — When `init` is run on a project that already has a `config.yaml` missing some default blocks (e.g., an older init that didn't include kanban defaults), should init merge in missing defaults or leave the file untouched? Currently `--force-config` overwrites entirely.
